# Shepherd Premium Single-User Roadmap

> **Purpose:** a durable, implementation-ready plan for making Shepherd feel dependable,
> polished, and premium for one operator. This document intentionally avoids team/RBAC,
> approval chains, billing administration, and other enterprise theater.
>
> **Status:** forward plan. Implement one numbered initiative at a time.
>
> **Companion documents:** [architecture.md](architecture.md),
> [deployment.md](deployment.md), [design-tokens.md](design-tokens.md), and the broader
> [platform roadmap](roadmap.md).

---

## 1. Product thesis

For a single-user operations product, “premium” means:

- **Nothing important is easy to lose.** Backups are complete, encrypted, verified, and
  restorable.
- **The system explains itself.** Health, failures, capacity, and recovery actions are
  visible without reading logs.
- **Past work remains useful.** Sessions, outcomes, diffs, commits, and decisions are
  searchable after the terminal closes.
- **Workspaces are deliberate.** Pens and layouts can be saved, restored, and moved
  between browsers.
- **Attention is respected.** Notifications are actionable, deduplicated, and quiet by
  default.
- **Maintenance is safe.** Upgrades, diagnostics, retention, and destructive actions
  have previews and recovery paths.
- **Power is discoverable.** Keyboard navigation is fast, but important actions never
  depend on memorizing shortcuts.

The goal is operational confidence, not feature count. Every initiative below must make
an important user outcome faster, safer, or easier to understand.

## 2. Current-state constraints

Shepherd does **not** currently have a complete backup/restore facility. Durable state is
split across several boundaries:

| State                                                                                                      | Current location                                 | Backup implication                                                                                           |
| ---------------------------------------------------------------------------------------------------------- | ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------ |
| Nodes, projects, sessions, events, users, audit data, encrypted credential envelopes, server-owned layouts | PostgreSQL                                       | Database must be captured consistently and restored through migrations.                                      |
| Credential decryption key                                                                                  | `FLOCK_MASTER_KEY` or `secrets/flock_master_key` | A database restore without the matching key cannot decrypt SSH credentials.                                  |
| Database password and deployment configuration                                                             | Docker secrets, `.env`, Compose overrides        | Needed to reconstruct a deployment, but some values should be regenerated rather than blindly restored.      |
| TLS certificates                                                                                           | Traefik ACME volume                              | Usually regenerated by ACME; should not be in the default portable archive.                                  |
| Session PTYs and live node processes                                                                       | `agentd` on each node                            | Not fully captured by an orchestrator backup; restore must clearly distinguish metadata from live processes. |
| Node-local transcripts/runtime artifacts                                                                   | Node filesystem / agent integration paths        | May require an optional node archive later; never imply they are included when they are not.                 |
| Node order, sidebar state, presets, and remaining UI preferences                                           | Browser `localStorage`                           | Must be promoted to a versioned server-side preferences document or explicitly exported.                     |

### Non-negotiable backup rules

1. Never call an archive “complete” unless its manifest states exactly what is included.
2. Never place plaintext credentials or the raw master key in an unencrypted archive.
3. Never restore over a running installation without a preflight, confirmation, and a
   rollback snapshot.
4. Never silently downgrade schema or archive versions.
5. A backup is not healthy until it has passed structural and cryptographic verification.
6. Automated restore testing must use an isolated database; never test against production.

## 3. How to use this plan

- Work initiatives in order unless the dependency table explicitly allows otherwise.
- One initiative should normally be one epic made of small PRs; each task can be its own
  PR.
- Add the failing test before implementation where practical.
- Keep wire contracts in `packages/shared` and validate all external input with Zod.
- Add database changes through forward-only Drizzle migrations.
- Mark a task complete only after its definition of done and validation checklist pass.
- Update this document in the implementation PR; record material architecture decisions
  in `docs/decisions/`.

## 4. Global definition of done

Every implementation initiative must satisfy the relevant gates:

```bash
pnpm typecheck
pnpm lint
pnpm test:unit
pnpm test:int
pnpm build
pnpm test:e2e                  # required for user-facing workflows
cd agentd && go test -race ./...  # required when agentd changes
```

Additionally:

- New routes require authentication/authorization and negative tests.
- New destructive actions require a confirmation surface and an audit event.
- New background jobs require idempotency, structured logging, retry limits, and visible
  last-success/last-error state.
- New persisted documents require a version field, tolerant reader, migration strategy,
  and size limit.
- Empty, loading, partial, stale, offline, success, and failure states must be designed.
- Accessibility: keyboard operation, visible focus, semantic labels, and no color-only
  status communication.
- Documentation must explain operator recovery, not only the happy path.

## Initiative 0 — Persistence inventory and server-owned preferences

### Why

Backup cannot be correct while important state is implicit or browser-local. This
initiative establishes a machine-readable inventory and moves meaningful workspace
preferences to the server. Ephemeral presentation state—open menus, scroll offsets,
temporary dialog input—should remain local and should not pollute backups.

### Scope

- Classify every persisted value as system record, encrypted secret, workspace
  preference, cache, runtime-only state, or external/node-local artifact.
- Add one versioned per-user preferences document for meaningful UI state.
- Retain a small browser cache for fast startup and offline tolerance.
- Document exclusions and ownership.

### Tasks

- [ ] **P0.1 — Persistence inventory.** Create a table mapping every Postgres table,
      Docker secret, environment variable, volume, localStorage key, and node-local artifact
      to owner, sensitivity, retention, backup inclusion, and restore behavior.
- [ ] **P0.2 — Preference contract.** Add `UserPreferencesV1Schema` in
      `packages/shared` with bounded arrays/strings and explicit defaults. Include node order,
      sidebar preference, layout presets, notification preferences, and other durable user
      choices. Exclude transient navigation and live terminal state.
- [ ] **P0.3 — Durable storage.** Add a `user_preferences` table or versioned JSONB column
      keyed by user, plus authenticated GET/PUT endpoints with optimistic concurrency
      (`revision` or ETag).
- [ ] **P0.4 — Browser migration.** On first successful load, import recognized legacy
      localStorage keys once, validate them, write them server-side, and mark migration
      complete. Invalid values fall back safely and are logged without user data.
- [ ] **P0.5 — Query/store integration.** Treat TanStack Query as server state and keep
      Zustand UI-only. Use localStorage only as a last-known cache, never as the authority.
- [ ] **P0.6 — Data manifest.** Add a generated/validated backup inventory module used
      by backup code and documented for operators.

### Definition of done

- A new browser receives the same meaningful workspace preferences after login.
- Every durable state category has explicit backup and restore semantics.
- No feature writes a new localStorage key without being classified in the inventory.
- Corrupt or future-version preferences do not prevent the shell from loading.

### Tests and validation

- Unit: schema defaults, bounds, unknown fields, migration from each legacy key.
- Integration: preferences ownership, revision conflicts, malformed payload rejection.
- E2E: set node order/layout on browser A; sign in on clean browser B; verify parity.
- Manual: clear localStorage and confirm no meaningful configuration is lost.

## Initiative 1 — Shepherd Vault backup foundation

### Why

The first backup release must be safe enough to trust. It should favor a versioned,
portable archive over environment-specific volume copying while supporting an operator
CLI before adding a polished UI.

### Product model

A `.flockvault` archive is an authenticated encrypted container containing:

- `manifest.json`: format version, Shepherd version, schema/migration version, created time,
  installation ID, inclusion/exclusion list, counts, checksums, and required capabilities.
- A consistent PostgreSQL logical dump (`pg_dump` custom format or an equivalently
  restorable logical representation).
- The master key and required application secrets, only inside the encrypted envelope.
- Sanitized deployment metadata sufficient to guide reconstruction.
- Server-owned user preferences.
- Optional node-artifact descriptors; actual node artifacts are excluded in v1 unless
  explicitly implemented and verified.

Use an established password-based encryption construction such as Argon2id for key
derivation plus an audited AEAD. Do not invent crypto or expose archive passwords to
process lists/logs.

### Tasks

- [ ] **B1.1 — Archive specification and threat model.** Write an ADR covering archive
      layout, encryption, KDF parameters, streamability, max sizes, tamper detection,
      password handling, key rotation, and explicit exclusions.
- [ ] **B1.2 — Shared manifest contract.** Add versioned Zod schemas and compatibility
      rules. Archive readers reject unsupported major versions and tolerate additive minor
      fields.
- [ ] **B1.3 — Consistent capture service.** Implement a bounded, cancellable backup job
      that obtains a logical database snapshot without prolonged application downtime.
- [ ] **B1.4 — Secret capture.** Read the active master key through the existing keyring
      abstraction. Never persist plaintext intermediate files; use restrictive temporary
      permissions and guaranteed cleanup.
- [ ] **B1.5 — Streaming archive writer.** Stream compression, encryption, and checksum
      generation to avoid holding the database or archive in memory.
- [ ] **B1.6 — Verification command.** Add a CLI operation that validates header,
      password, AEAD tag, checksums, manifest contract, expected files, and logical dump
      readability without restoring.
- [ ] **B1.7 — Operator CLI.** Add documented create/list/verify commands suitable for
      Compose deployments. Password input must come from an interactive prompt or protected
      file descriptor, not a command argument.
- [ ] **B1.8 — Settings UI.** Add Settings → Backups with Create Backup, progress,
      cancellation, download, last verified time, contents summary, and clear exclusions.
- [ ] **B1.9 — Audit and observability.** Record backup start/success/failure/verify events
      without archive paths, passwords, keys, or secret values in logs.

### Failure behavior

- A failed job leaves no valid-looking partial archive; use a temporary name and atomic
  finalization.
- Insufficient disk, lost database connection, cancellation, or key-read failure produces
  a specific error and cleans temporary data.
- The UI remains usable during backup and cannot start duplicate jobs accidentally.

### Definition of done

- An operator can create and verify an encrypted archive from CLI and UI.
- The archive contains everything declared by the manifest and nothing undeclared.
- Wrong passwords, corruption, truncation, and unsupported versions fail safely.
- No plaintext database dump, master key, or credential appears on disk after completion.

### Tests and validation

- Unit: manifest compatibility, KDF/encryption test vectors, corruption/truncation cases,
  redaction, filename/path safety.
- Integration: seed every table and encrypted credential; create; verify; inspect counts.
- Security: scan temporary directories, logs, archive strings, and process arguments for
  known seeded secrets.
- E2E: create backup, observe progress, download, verify success and failure messages.
- Performance: representative large event history with bounded memory and documented
  duration/size.

## Initiative 2 — Safe restore and Recovery Center

### Why

A backup feature without a proven restore path creates false confidence. Restore is a
maintenance operation, not a normal API mutation, and must be designed around preflight,
isolation, rollback, and explicit handling of live sessions.

### Tasks

- [ ] **R2.1 — Restore state machine.** Define `uploaded → verified → preflighted →
snapshot-created → restoring → validating → complete|rolled-back|failed`. Persist job
      state so a browser refresh cannot orphan the operation.
- [ ] **R2.2 — Preflight service.** Check archive version, Shepherd compatibility, schema
      direction, free disk, database connectivity, master-key presence, required tools,
      installation identity, conflicts, and node/session implications.
- [ ] **R2.3 — Restore preview.** Show archive date/version, record counts, included
      secrets/preferences, excluded node artifacts, destructive effects, and whether active
      sessions will remain alive, become detached, or be marked stale.
- [ ] **R2.4 — Automatic rollback snapshot.** Before mutation, create and verify a local
      pre-restore vault. Refuse restore if this cannot be completed unless the operator uses
      an explicit documented break-glass CLI flag.
- [ ] **R2.5 — Isolated restore engine.** Restore first into a temporary database, run
      migrations and integrity checks, then perform a controlled cutover. Never stream
      unvalidated SQL into the active database.
- [ ] **R2.6 — Secret validation.** Confirm restored encrypted credential envelopes can be
      decrypted by the restored master key using non-sensitive probes.
- [ ] **R2.7 — Post-restore reconciliation.** Reconnect nodes, reconcile projects and live
      sessions, invalidate auth sessions as appropriate, refresh caches, and produce a
      concise recovery report.
- [ ] **R2.8 — Recovery Center UI.** Add preflight checklist, confirmation phrase,
      progress, recovery report, retry guidance, and rollback action.
- [ ] **R2.9 — Disaster-recovery CLI.** Support restore when the web application cannot
      boot, with the same verifier and engine used by the UI.
- [ ] **R2.10 — Runbook.** Document new-host recovery, lost database, lost master key,
      wrong Shepherd version, interrupted restore, and rollback.

### Definition of done

- A vault restores into a clean installation and reproduces declared durable state.
- A failed restore returns the prior installation to a verified working state.
- Restore never claims to resurrect a live PTY or node artifact it did not capture.
- The application cannot serve normal mutations during final cutover.

### Tests and validation

- Integration matrix: same version, newer app/older archive, unsupported newer archive,
  wrong password, corrupt dump, missing entry, credential decrypt failure.
- Failure injection at every state transition; assert rollback and resumable reporting.
- E2E: upload → preview → confirm → restore → forced re-auth → verify data/preferences.
- Disaster drill: destroy a disposable Compose stack and rebuild it only from documented
  prerequisites plus a vault.

## Initiative 3 — Scheduled backups and restore drills

### Why

Manual backups decay. Premium protection makes successful, recent, independently
verified backups the default and makes failures visible without becoming noisy.

### Tasks

- [ ] **A3.1 — Backup destination abstraction.** Start with local filesystem and add an
      S3-compatible destination later. Credentials use the encrypted secret store.
- [ ] **A3.2 — Scheduler and lease.** Persist schedules, use a single-run lease, handle
      restarts/time changes, and prevent overlapping jobs.
- [ ] **A3.3 — Retention policy.** Support simple daily/weekly retention with protected
      manual backups, dry-run preview, and never delete the last verified backup.
- [ ] **A3.4 — Destination health.** Test write/read/delete permissions and display free
      space or remote errors.
- [ ] **A3.5 — Automatic verification.** Verify every completed upload by reading it back;
      only then mark it healthy.
- [ ] **A3.6 — Isolated restore drill.** Periodically restore the latest vault into a
      temporary database, run integrity probes, then destroy it. Never connect to nodes or
      send notifications from the drill environment.
- [ ] **A3.7 — Backup health card.** Show latest verified backup, age, destination,
      duration, size, next run, drill result, and actionable errors.
- [ ] **A3.8 — Alerts.** Notify only when backup age exceeds policy, consecutive runs fail,
      verification fails, or destination capacity is low.

### Definition of done

- Backups run unattended, enforce retention safely, and visibly report health.
- A read-back verification and isolated restore drill can detect an unusable archive.
- Scheduler behavior is deterministic across restarts and daylight-saving changes.

### Tests and validation

- Fake-clock scheduler tests; lease/race tests; retry/backoff tests.
- Integration against local filesystem and an S3-compatible test service if supported.
- Retention property tests, especially “never delete last verified.”
- E2E for schedule creation, health, history, and failure recovery.

## Initiative 4 — Session history and universal search

### Why

Closed agent sessions should become institutional memory for the individual operator,
not disappear into a terminal buffer. Search should answer: “Which agent changed this
file?”, “What solved this error?”, and “Where was this commit produced?”

### Tasks

- [ ] **H4.1 — History contract.** Define a session summary: task/note, node, project,
      agent, timestamps, duration, outcome, final status, files changed, commits, branch,
      token/cost data when available, and artifact availability.
- [ ] **H4.2 — Durable close snapshot.** On session close, asynchronously compute and
      persist a bounded summary. Failure must not block termination.
- [ ] **H4.3 — Transcript policy.** Decide which normalized transcript/activity content
      is durable, searchable, redacted, size-bounded, and deletable. Never index secret
      values or raw environment data.
- [ ] **H4.4 — Search index.** Begin with PostgreSQL full-text/trigram indexes. Define a
      provider interface before introducing an external search service.
- [ ] **H4.5 — Search API.** Typed query, filters, pagination/cursor, snippets, result
      categories, and strict result caps/timeouts.
- [ ] **H4.6 — History page.** Project-grouped timeline/table with filters for status,
      agent, node, date, files, and Git outcome.
- [ ] **H4.7 — Session detail/replay.** Read-only chronological activity with task,
      important events, files, diff/commits, terminal transcript where retained, and clear
      missing-artifact messaging.
- [ ] **H4.8 — Universal search UI.** Search nodes, projects, sessions, history, changed
      files, commits, and commands from one surface with keyboard navigation.

### Definition of done

- Closed sessions remain discoverable and explain their useful outcome.
- Search results are permission-safe, fast on representative data, and link to durable
  destinations.
- Missing transcripts or Git repositories degrade honestly rather than breaking pages.

### Tests and validation

- Unit: indexing normalization/redaction, ranking, snippet safety, duration/outcome.
- Integration: close session → summary/index → filtered search and pagination.
- E2E: search a known task/file/commit and open its history detail.
- Performance: define and meet a target (initial recommendation: p95 under 300 ms for
  100,000 indexed session artifacts on reference hardware).

## Initiative 5 — Global activity timeline

### Why

Shepherd already exposes session-level activity and an audit log. A premium global timeline
should unify meaningful operational events without duplicating raw telemetry or becoming
an unreadable firehose.

### Tasks

- [ ] **T5.1 — Canonical activity projection.** Project existing event/audit/session/node
      data into a read model with category, severity, actor/source, target, timestamp, and
      stable deep link. Do not duplicate source-of-truth rows unnecessarily.
- [ ] **T5.2 — Event taxonomy.** Include session lifecycle/attention/outcome, node
      connection/health, project Git milestones, backups/restores, updates, and important
      settings changes. Exclude routine heartbeats and noisy status churn.
- [ ] **T5.3 — Timeline API.** Cursor pagination, category/project/node filters, time
      bounds, and aggregated repeat events.
- [ ] **T5.4 — Timeline UI.** Human-readable grouped timeline with icons, relative and
      exact times, filters, deep links, empty state, and live insertion without scroll jumps.
- [ ] **T5.5 — Explainability.** Every warning/error states impact and recommended action;
      raw detail is secondary and copyable.

### Definition of done

- The operator can reconstruct important system activity without reading server logs.
- Repeated disconnects/errors are grouped while preserving counts and time range.
- Every displayed event has a reliable source and destination or explicitly says none.

### Tests and validation

- Projection unit tests for every event category and redaction rule.
- Integration pagination/order tests with identical timestamps and concurrent inserts.
- E2E filters, live updates, deep links, and grouped-repeat expansion.

## Initiative 6 — Project health center

### Why

Git visibility belongs to projects. The operator should understand repository state and
agent contention without opening a terminal or managing Git on the agents’ behalf.

### Tasks

- [ ] **G6.1 — Health snapshot contract.** Branch, upstream, dirty/staged/untracked
      counts, ahead/behind, merge/rebase state, recent commit, detected conflicts, active
      agents, and refresh timestamp.
- [ ] **G6.2 — Bounded collector.** Reuse the existing Git service, add timeouts and
      concurrency limits, cache briefly, and never run expensive repository-wide operations
      on every render.
- [ ] **G6.3 — Agent contention signals.** Show multiple active agents on the same project
      and overlapping changed files when known. Inform; do not seize control of worktrees or
      Git state.
- [ ] **G6.4 — Project overview.** Prominent Source Control entry, health summary,
      changed-file preview, recent commits, active agents, and last refresh/error.
- [ ] **G6.5 — Tests signal.** Add an optional, explicitly configured test command or
      consume agent-reported results. Never auto-run arbitrary repository commands merely
      by viewing a page.
- [ ] **G6.6 — Staleness model.** Distinguish fresh, cached, unreachable node, timeout,
      non-Git directory, and permission failure.

### Definition of done

- Project pages answer “what branch/state is this in, what changed, and who is working?”
- Collection cannot overload a node with many projects.
- Shepherd remains observational: no implicit checkout, reset, worktree creation, or merge.

### Tests and validation

- Unit fixtures for clean/dirty/ahead/behind/conflict/detached/non-repo states.
- Integration against real temporary Git repositories and timeout/error cases.
- E2E project health rendering, refresh, Git page navigation, and stale messaging.

## Initiative 7 — Node health history and capacity

### Why

Current CPU/memory/storage snapshots are useful but transient. History and capacity
signals let the operator distinguish a momentary spike from a deteriorating node.

### Tasks

- [ ] **N7.1 — Metrics contract.** Version node samples for CPU, memory, filesystem,
      load, uptime, agent count, agentd version, connection latency, and collection errors.
- [ ] **N7.2 — Sampling and rollups.** Keep collection off the live PTY path; apply
      bounded concurrency, retention, and hourly/daily rollups to control database growth.
- [ ] **N7.3 — Health rules.** Conservative configurable thresholds for disk, memory,
      sustained load, repeated disconnects, clock skew, stale agentd, and unreachable state.
- [ ] **N7.4 — Node detail charts.** Current values plus 1h/24h/7d trends, capacity,
      disconnect history, active projects/agents, agentd version, and precise collection age.
- [ ] **N7.5 — Paddock summaries.** Keep cards calm: only show actionable warnings and
      key current metrics; detail belongs on the node page.
- [ ] **N7.6 — Storage forecast.** After sufficient samples, estimate time-to-full with a
      confidence qualifier. Suppress unstable forecasts.

### Definition of done

- A node’s present and recent health is understandable from one page.
- Metrics retention remains bounded and documented.
- Threshold breaches require persistence/hysteresis and do not flap on single samples.

### Tests and validation

- Unit: rollups, retention, threshold hysteresis, missing data, forecast confidence.
- Integration: ingestion/upsert, concurrent samples, cleanup, disconnected node.
- E2E: node card warning → node detail → chart/incident explanation.
- Load: expected node count and sampling interval with measured DB growth.

## Initiative 8 — Workspace snapshots

### Why

Pens organize live agents, but users also need intentional named arrangements: “release
work,” “frontend pair,” or “incident.” A snapshot should restore layout and selection
without pretending it can resurrect terminated agents.

### Tasks

- [ ] **W8.1 — Snapshot contract.** Versioned project/node context, Pen definitions,
      membership, arrangement, order, zoom/focus preference, and optional open tool/page.
- [ ] **W8.2 — Server persistence.** CRUD endpoints, names/descriptions, updated time,
      duplicate, and last-used metadata.
- [ ] **W8.3 — Capture UX.** “Save workspace” previews exactly what will be stored and
      distinguishes live references from durable configuration.
- [ ] **W8.4 — Restore planner.** Classify referenced sessions as available, closed,
      missing, or moved; preview the resulting layout before apply.
- [ ] **W8.5 — Safe apply.** Reconcile existing Pens without duplicating membership;
      preserve unrelated sessions and provide undo for the current browser session.
- [ ] **W8.6 — Snapshot library.** Searchable list with project, agent count, last used,
      duplicate, rename, delete confirmation, and command-palette actions.

### Definition of done

- A named layout restores consistently on another browser.
- Missing/closed sessions produce a useful partial restore, not failure or blank panes.
- Applying the same snapshot twice is idempotent.

### Tests and validation

- Unit: version parsing, reconciliation, missing sessions, idempotency, undo state.
- Integration: ownership and CRUD conflicts.
- E2E: save four-agent Pen layout; alter it; restore; repeat on clean browser.

## Initiative 9 — Smart notifications

### Why

Notifications should bring the operator back only when action or awareness is valuable.
Shepherd already has web-push foundations; this initiative turns them into a coherent,
user-controlled attention system.

### Tasks

- [ ] **Q9.1 — Notification taxonomy.** Agent needs input, agent finished, agent failed,
      node disconnected/recovered, sustained node capacity warning, backup unhealthy,
      restore complete/failed, and update result.
- [ ] **Q9.2 — Preference model.** Per-category channel and severity, quiet hours/time
      zone, node/project mute, browser-active suppression, and test notification.
- [ ] **Q9.3 — Deduplication state machine.** Stable incident keys, cooldowns,
      escalation rules, and recovery notifications. Never notify every metric sample.
- [ ] **Q9.4 — Actionable payloads.** Deep link to the exact agent/project/node/recovery
      page; concise title, impact, and next action without sensitive terminal content.
- [ ] **Q9.5 — Notification center.** Recent notifications, read/dismiss, mute action,
      source health, permission status, and troubleshooting.
- [ ] **Q9.6 — Delivery observability.** Record enqueue/sent/failure/expired subscription
      without storing private payload content unnecessarily.

### Definition of done

- Default notifications are low-noise and immediately actionable.
- Quiet hours, mutes, active-browser suppression, dedupe, and recovery work reliably.
- Browser permission denial does not degrade in-app notification behavior.

### Tests and validation

- Fake-clock state-machine tests for quiet hours, cooldown, recovery, and time zones.
- Integration push subscription expiration/retry and preference enforcement.
- E2E in-app notification center and deep links; manual web-push validation on supported
  desktop and mobile browsers.

## Initiative 10 — Command palette and keyboard discovery

### Why

Shepherd already supports Command/Ctrl+K. Premium keyboard UX requires a complete command
model, recency, context-aware actions, and visible discovery—not a hidden shortcut.

### Tasks

- [ ] **K10.1 — Command registry.** One typed registry for navigation and actions with
      stable IDs, labels, keywords, icons, shortcuts, availability predicate, and execution.
- [ ] **K10.2 — Search providers.** Nodes, projects, live agents, history, workspace
      snapshots, settings, and actions. Support category prefixes only as optional power UX.
- [ ] **K10.3 — Ranking.** Fuzzy score plus recency/frequency and current-context boost;
      deterministic tie-breaking and no surprising hidden personalization.
- [ ] **K10.4 — Context actions.** On a project/node/agent, surface the few meaningful
      actions first. Destructive actions still require confirmation outside the palette.
- [ ] **K10.5 — Discovery.** Keep a compact header search affordance, show shortcuts in
      menus/tooltips, and add a `?` keyboard-shortcut reference overlay.
- [ ] **K10.6 — Navigation shortcuts.** Define a small conflict-safe set for Paddock,
      project Git, next attention item, active Pen, and panel focus. Never intercept terminal
      keystrokes unless the modifier contract is unambiguous.

### Definition of done

- Any node/project/agent/history item is reachable within a few keystrokes.
- Commands accurately reflect current availability and never fail silently.
- All power actions remain discoverable with mouse/touch.

### Tests and validation

- Unit: registry uniqueness, ranking, recency, predicate behavior, terminal exclusions.
- Component: keyboard navigation, focus trap, screen-reader labels, no-results states.
- E2E: Command/Ctrl+K navigation and actions from overview, agent terminal, and mobile.

## Initiative 11 — Update Center

### Why

Showing the version is helpful; a premium product also explains update availability,
compatibility, migrations, node-daemon rollout, and recovery. Shepherd should not silently
self-update by default.

### Tasks

- [ ] **U11.1 — Version manifest.** Current web/orchestrator/agentd/schema versions and
      build commit from one trusted endpoint.
- [ ] **U11.2 — Release provider.** Fetch signed/trusted release metadata with timeout,
      caching, channel selection, and no hard dependency on GitHub availability.
- [ ] **U11.3 — Compatibility/preflight.** Display database migration needs, agentd
      rollout, browser image requirements, disk space, and backup freshness.
- [ ] **U11.4 — Update Center UI.** Current/latest, release notes link, channel, check
      time, preflight, documented update command, migration state, and node rollout status.
- [ ] **U11.5 — Backup gate.** Recommend or require a recent verified vault for
      migration-bearing upgrades; allow explicit CLI override.
- [ ] **U11.6 — Assisted update (later).** Only after the informational center is mature,
      consider an opt-in deployment-specific updater with signed artifacts, health checks,
      and rollback. Do not assume Shepherd can safely mutate every Compose deployment.

### Definition of done

- The operator knows what is installed, what is available, and what an update will do.
- Update checks fail quietly and never block normal Shepherd use.
- No update is represented as reversible unless rollback is actually tested.

### Tests and validation

- Unit: semantic versions/channels, signed metadata, cache/timeout, incompatible release.
- Integration: version endpoint and migration/agentd status.
- E2E: current, update available, offline, incompatible, and backup-required states.

## Initiative 12 — Diagnostics and support bundle

### Why

When something fails, users need a fast way to understand it or share useful evidence.
Diagnostics must be transparent and aggressively redact credentials, paths, terminal
content, hostnames, and user data by default.

### Tasks

- [ ] **D12.1 — Diagnostic checks.** Database/migration, disk, Docker, Traefik reachability,
      orchestrator/web versions, node connectivity, agentd version, clock skew, browser
      image, backup health, push status, and recent bounded error counts.
- [ ] **D12.2 — Structured check contract.** Stable ID, status, summary, impact,
      remediation, timestamp, duration, and optional sanitized detail.
- [ ] **D12.3 — Redaction library.** Central allowlist-first sanitizer for secrets,
      tokens, cookies, authorization headers, URLs, usernames, home paths, IPs/hostnames, and
      repository names. Avoid regex-only confidence; prefer structured collection.
- [ ] **D12.4 — Previewable bundle.** Show every file/category and redaction level before
      export. Default bundle excludes transcripts, diffs, environment values, SSH config,
      database content, and raw logs.
- [ ] **D12.5 — Export.** Bounded ZIP with manifest/checksums and optional password
      encryption. Generate asynchronously with expiry and guaranteed deletion.
- [ ] **D12.6 — Diagnostics UI.** Run all/retry, copy remediation, download preview, and
      link to relevant settings or node/project pages.

### Definition of done

- Common deployment and node failures produce an actionable diagnosis.
- Seeded secrets and private values do not appear in the default bundle.
- The operator can inspect exactly what will leave the installation.

### Tests and validation

- Unit: adversarial redaction corpus including encoded/embedded tokens and URLs.
- Integration: each diagnostic failure mode and timeout; bundle expiry/cleanup.
- Security test: seed canary secrets across DB/env/logs and assert archive contains none.
- E2E: run diagnostics, expand result, preview bundle, export.

## Initiative 13 — Data ownership and retention

### Why

Premium software makes storage legible and gives the user control over history without
turning settings into database administration.

### Tasks

- [ ] **O13.1 — Storage inventory UI.** Database size by category, event/history growth,
      backup storage, browser artifacts, and node-local data where measurable. Show “unknown”
      honestly when remote measurement is unavailable.
- [ ] **O13.2 — Retention policies.** Simple presets plus custom limits for events,
      closed-session transcripts, activity, diagnostics, notifications, metrics rollups, and
      backups. Preserve essential relational records and audit/security minimums.
- [ ] **O13.3 — Dry-run cleanup.** Preview counts, date ranges, approximate bytes, and
      consequences before deletion.
- [ ] **O13.4 — Idempotent cleanup jobs.** Batches, cancellation boundaries, progress,
      retry, vacuum guidance, audit entries, and no live-path blocking.
- [ ] **O13.5 — Scoped exports.** Export a project/session history with manifest,
      metadata, commits/diff references, and explicitly selected transcript content.
- [ ] **O13.6 — Destructive deletion.** Delete session history/project metadata/all user
      data with typed confirmation, dependency preview, verified completion report, and
      recommendation to create a backup first.
- [ ] **O13.7 — Privacy defaults.** Document what Shepherd records, where it resides, how
      long it remains, and what is never sent externally.

### Definition of done

- The operator can understand and control Shepherd’s storage growth.
- Cleanup previews match actual results within documented estimation limits.
- Destructive operations are auditable, resumable where possible, and cannot delete live
  sessions accidentally.

### Tests and validation

- Unit: retention boundaries, protected records, dependency planning, size estimates.
- Integration: large batched cleanup, cancellation/retry, FK integrity, concurrent writes.
- E2E: storage view → cleanup preview → confirm → completion report.

## 6. Cross-cutting architecture

### Background jobs

Backup, restore, rollups, indexing, cleanup, and diagnostics need a common durable job
model rather than six bespoke polling implementations. Before the second subsystem needs
it, introduce a small job framework with:

- typed job kind and versioned payload;
- queued/running/succeeded/failed/cancelled state;
- progress phase and bounded public message;
- lease/heartbeat and orphan recovery;
- idempotency key;
- attempts and bounded retry policy;
- created/started/finished timestamps;
- user-safe error code plus server-side structured detail;
- retention/cleanup;
- SSE or WebSocket progress fan-out with polling fallback.

Do not turn this into a general workflow engine. Build only the primitives needed by the
first two consumers.

### Security and privacy

- Archive and diagnostic downloads use short-lived, single-user authorization and expire.
- Sensitive jobs never put secrets in payload JSON, database errors, telemetry, or audit
  detail.
- External destinations are opt-in and communicate exactly what leaves Shepherd.
- Search/transcript storage gets explicit retention and redaction from its first release.
- Use constant-time comparisons and established crypto libraries for vault verification.

### UX conventions

- Use the existing design tokens and calm black/gray dark surfaces; retain blue for
  selection/action emphasis.
- Put overview signals on Paddock cards and detailed diagnostics on node/project pages.
- Prefer one clear status sentence plus a next action over dashboards full of badges.
- Show freshness everywhere health data appears.
- Every long operation survives navigation and browser refresh.
- Every empty state explains whether nothing exists, data is loading, data is filtered,
  or the source is unavailable.

## 7. Release discipline

For each initiative:

1. Land contracts and migrations first.
2. Land backend behavior with integration tests and no UI dependency.
3. Land the smallest complete UI workflow.
4. Validate against a production-like Compose stack and at least one real remote node
   when node behavior is involved.
5. Add observability/runbooks.
6. Ship behind a temporary feature flag only when live-path risk warrants it; remove the
   flag after validation.
7. Update this plan with completion date, key decisions, and deferred follow-ups.

## 8. Program-level success criteria

This roadmap is complete when one operator can:

- rebuild a destroyed Shepherd installation from a recently verified encrypted vault;
- see that backups and restore drills are healthy before disaster;
- search completed work by task, file, project, agent, or commit;
- understand project Git state and node capacity without opening a terminal;
- save and restore useful Pen arrangements across browsers;
- receive low-noise, actionable notifications with quiet hours;
- navigate the system quickly without already knowing hidden shortcuts;
- understand installed/update versions and upgrade prerequisites;
- generate a transparent, sanitized diagnostic bundle;
- understand, export, retain, and delete their own data safely.

The qualitative bar is equally important: failures explain impact and recovery, stale
data identifies itself, destructive operations have previews, and no feature claims a
guarantee that its implementation cannot prove.
